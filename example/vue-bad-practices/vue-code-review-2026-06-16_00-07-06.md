## Issues Found

1. **Options API en lugar de Composition API con `<script setup>`**
    - **Severity**: Medium
    - **Description**: Ambos componentes (`Todo.vue` y `TodoItem.vue`) usan `defineComponent` con Options API (`data()`, `methods`). Las guidelines exigen usar `<script setup>` con Composition API para mayor concisión, mejor inferencia de tipos TypeScript y rendimiento en tiempo de compilación. El uso de Options API genera más boilerplate, reduce la colocación lógica y dificulta el tree-shaking de código no utilizado.
    - **Location**: [src/components/Todo.vue:24-57](src/components/Todo.vue#L24-L57), [src/components/TodoItem.vue:8-29](src/components/TodoItem.vue#L8-L29)
    - **Recommendation**: Migrar ambos componentes a `<script setup lang="ts">`. Reemplazar `data()` con `ref`/`reactive`, y `methods` con funciones regulares declaradas en el bloque `<script setup>`.

2. **`<template>` ubicado antes de `<script>`**
    - **Severity**: Low
    - **Description**: En ambos SFCs el bloque `<template>` aparece antes del bloque `<script>`. La convención establecida en las guidelines indica que `<script>` debe preceder a `<template>` para permitir leer la lógica y dependencias antes de revisar el HTML, mejorando la legibilidad general del componente.
    - **Location**: [src/components/Todo.vue:1](src/components/Todo.vue#L1), [src/components/TodoItem.vue:1](src/components/TodoItem.vue#L1)
    - **Recommendation**: Mover el bloque `<script>` al inicio del archivo, antes de `<template>`.

3. **Nombres de variables, props y métodos no descriptivos**
    - **Severity**: Medium
    - **Description**: El código usa identificadores de una sola letra o sin significado: `n` (input del usuario), `stuff` (lista de todos), `t` (iterador en `v-for`), `i` (índice), `v` (valor local en `addOne`), `x` (prop en `TodoItem`), `a` y `b` (métodos en `TodoItem`). Estos nombres hacen el código ilegible e inmantenible; cualquier colaborador debe leer toda la implementación para entender el propósito de cada pieza.
    - **Location**: [src/components/Todo.vue:6, 36, 44, 14](src/components/Todo.vue#L6), [src/components/TodoItem.vue:2-4, 13, 21, 25](src/components/TodoItem.vue#L2)
    - **Recommendation**: Renombrar siguiendo las convenciones: `n` → `newTodoText`, `stuff` → `todos`, `t` → `todo`, `i` → `index`, `v` → `trimmedValue`, `x` (prop) → `todo`, `a` → `handleToggle`, `b` → `handleDelete`.

4. **Nombre del componente `todoitem` no sigue PascalCase**
    - **Severity**: Medium
    - **Description**: La propiedad `name` del componente está definida como `'todoitem'` (todo en minúsculas). Las guidelines exigen PascalCase para nombres de componentes. Además, el nombre incorrecto puede causar advertencias en Vue DevTools y dificulta la identificación del componente en el árbol de componentes.
    - **Location**: [src/components/TodoItem.vue:12](src/components/TodoItem.vue#L12)
    - **Recommendation**: Cambiar `name: 'todoitem'` a `name: 'TodoItem'`. Al migrar a `<script setup>`, el nombre se infiere automáticamente del nombre del archivo.

5. **`v-for` usa el índice como `:key`**
    - **Severity**: Medium
    - **Description**: Se usa `:key="i"` (el índice del array) como clave en el `v-for`. Cuando el array cambia (eliminación, reordenamiento), Vue reutiliza incorrectamente los nodos DOM porque las claves no están ligadas a la identidad del dato sino a su posición. Esto puede causar bugs de renderizado, animaciones incorrectas y pérdida de estado local en componentes hijos.
    - **Location**: [src/components/Todo.vue:15](src/components/Todo.vue#L15)
    - **Recommendation**: Usar un identificador único por ítem. Agregar un campo `id` (e.g., `crypto.randomUUID()` o un contador incremental) al objeto todo y usarlo como key: `:key="todo.id"`.

6. **Mutación directa de props en el método padre**
    - **Severity**: High
    - **Description**: El método `flipIt` en `Todo.vue` recibe el objeto todo por referencia y muta directamente su propiedad `x.done = !x.done`. Aunque en Vue 3 esto puede funcionar accidentalmente porque el objeto es reactivo, viola el principio fundamental de inmutabilidad de props: los componentes hijos no deben mutar datos que pertenecen al padre. Esto rompe el flujo de datos unidireccional, hace el código impredecible y puede causar bugs difíciles de rastrear en aplicaciones más complejas.
    - **Location**: [src/components/Todo.vue:49-51](src/components/Todo.vue#L49-L51)
    - **Recommendation**: En lugar de mutar el objeto directamente, el evento emitido desde `TodoItem` debe llevar el índice o id del ítem. El padre busca el ítem en su propio array y lo modifica: `const todo = todos.value.find(t => t.id === payload.id); if (todo) todo.done = !todo.done`.

7. **Props no tipadas con interfaz TypeScript**
    - **Severity**: Medium
    - **Description**: La prop `x` en `TodoItem.vue` define su tipo usando `Object as PropType<{ text: string; done: boolean }>` inline dentro de la definición de Options API. Las guidelines requieren definir una interfaz TypeScript separada para describir la forma del objeto de props, lo que mejora la reusabilidad del tipo, la legibilidad y la mantenibilidad.
    - **Location**: [src/components/TodoItem.vue:13-18](src/components/TodoItem.vue#L13-L18)
    - **Recommendation**: Al migrar a `<script setup>`, definir una interfaz separada y usarla con `defineProps<TodoItemProps>()`:
      ```typescript
      interface TodoItemProps {
        todo: Todo;
      }
      const props = defineProps<TodoItemProps>();
      ```

8. **`emits` no tipado con interfaz TypeScript**
    - **Severity**: Medium
    - **Description**: Los eventos se declaran como un array de strings `emits: ['flip', 'erase']` sin información sobre el tipo del payload. Las guidelines exigen una interfaz TypeScript dedicada para `defineEmits` que describa cada evento y su payload, garantizando type safety al emitir y consumir eventos.
    - **Location**: [src/components/TodoItem.vue:19](src/components/TodoItem.vue#L19)
    - **Recommendation**: Al migrar a `<script setup>`, definir una interfaz y usar `defineEmits<TodoItemEmits>()`:
      ```typescript
      interface TodoItemEmits {
        (event: 'toggleTodo', payload: { id: string }): void;
        (event: 'deleteTodo', payload: { id: string }): void;
      }
      const emit = defineEmits<TodoItemEmits>();
      ```

    - **Severity**: Low
    - **Description**: Los eventos emitidos se llaman `flip` y `erase`. Estos nombres no dejan claro qué acción representan en el dominio de la aplicación. Las guidelines piden nombres de eventos claros y descriptivos que comuniquen la intención del evento.
    - **Location**: [src/components/Todo.vue:17-18](src/components/Todo.vue#L17-L18), [src/components/TodoItem.vue:19](src/components/TodoItem.vue#L19)
    - **Recommendation**: Renombrar `flip` → `toggleTodo` y `erase` → `deleteTodo` para alinear con el vocabulario del dominio y las convenciones de las guidelines.

10. **Método `addOne` no usa el prefijo `handle`**
    - **Severity**: Low
    - **Description**: El método `addOne` es un event handler del `@submit.prevent` del formulario. Las guidelines establecen que los manejadores de eventos deben usar el prefijo `handle` (e.g., `handleSubmit`, `handleAddTodo`) para indicar claramente su propósito como respuesta a un evento del usuario.
    - **Location**: [src/components/Todo.vue:43](src/components/Todo.vue#L43)
    - **Recommendation**: Renombrar `addOne` → `handleAddTodo` y actualizar la referencia en el template: `@submit.prevent="handleAddTodo"`.

11. **Ausencia de separación de responsabilidades — sin composable para lógica de todos**
    - **Severity**: Medium
    - **Description**: `Todo.vue` mezcla la lógica de estado (lista de todos, agregar, eliminar, togglear) con la presentación del UI en un único componente. Las guidelines promueven la separación de responsabilidades: la lógica reactiva reutilizable debe extraerse a composables (`useTodos`), mientras que los componentes se enfocan en el renderizado. Esto hace que la lógica no sea testeable de forma aislada ni reutilizable.
    - **Location**: [src/components/Todo.vue:33-56](src/components/Todo.vue#L33-L56)
    - **Recommendation**: Extraer la lógica a un composable `src/composables/useTodos.ts` que exponga `todos`, `addTodo(text)`, `toggleTodo(id)` y `deleteTodo(id)`. `Todo.vue` consume el composable y delega toda lógica al mismo.

12. **Ausencia de tipo `Todo` definido como entidad de dominio**
    - **Severity**: Low
    - **Description**: No existe un archivo de tipos para la entidad `Todo`. El tipo `{ text: string; done: boolean }` se repite inline en múltiples lugares. Las guidelines exigen archivos `.type.ts` para los tipos compartidos entre componentes. Además, la ausencia de un campo `id` fuerza el uso del índice como key en `v-for` (ver Issue #5).
    - **Location**: [src/components/Todo.vue:36-39](src/components/Todo.vue#L36-L39), [src/components/TodoItem.vue:15](src/components/TodoItem.vue#L15)
    - **Recommendation**: Crear `src/types/todo.type.ts` con:
      ```typescript
      export interface Todo {
        id: string;
        text: string;
        done: boolean;
      }
      ```
      Importar y usar `Todo` en todos los componentes y el composable.
